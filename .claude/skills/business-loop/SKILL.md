# AI 變現商業模式主迴圈(Business Loop）

這是架在 `reels-script` 與 `threads-post` 兩個內容模型「之上」的**總指揮**。
內容模型負責「單篇怎麼寫」;本 skill 負責「整盤生意怎麼賺、每天做什麼、如何越跑越準」。

**每次啟動都必須先完整讀取同目錄的 `MODEL.md`。** 它是活的商業模型,權重最高的是
「六、已驗證的賺錢模式」與「五、目前 Offer 階梯」。

---

## 三種模式(先判斷使用者意圖)

- 使用者說「派工 / 安排 / 排行事曆 / 今天(這兩天)做什麼」→【派工模式】
- 使用者說「覆盤 / 回報數據 / 今天結果是… / 這篇成效」→【覆盤模式】
- 使用者說「週會 / 升級模型 / 重算 offer / 重新定位」→【週會模式】

---

## 派工模式(Dispatch)

目標:把 MODEL.md 的當前假設,轉成**今天/接下來兩天可執行的具體任務**,並寫進 Google 行事曆。

1. 讀 `MODEL.md`(尤其「四、迴圈協定」定義的每日節奏、「七、實驗待辦」的下一個實驗)。
2. 讀 `LOG.md` 最近 3 天,確認上一個實驗的狀態(進行中 / 待覆盤 / 已結案)。
3. 依「每日節奏模板」產出當日任務,每天固定四段:
   - **晨間派工**(黃金時段發文,套 threads/reels 模型)
   - **午間互動**(守留言、回同領域熱門貼文,衝對話速度)
   - **核心實驗**(當天推進「七、實驗待辦」最上面那一個)
   - **晚間覆盤**(回報數據到 LOG)
4. 用 Google Calendar MCP 建立事件(calendarId 用主行事曆、timeZone `Asia/Taipei`):
   - 標題前綴用 emoji 分類:🌅 派工 / 💬 互動 / 🧪 實驗 / 📊 覆盤。
   - description 寫清楚「怎麼做、賭什麼、要回報哪個數字」,並註明 `依 business-loop MODEL.md`。
   - colorId:派工=5(Banana)、互動=2(Sage)、實驗=11(Tomato)、覆盤=7(Peacock)。
5. 建完事件後,在對話裡條列「這兩天的重點與各自要驗證的假設」,並提醒晚上回來覆盤。
6. 不要一次排超過受眾/時間預算能負荷的量(見 MODEL.md「三、我的現況」的每日投入)。

## 覆盤模式(Measure → Upgrade)

目標:用真實數據校正模型,這是「自我升級」的核心。

1. 收使用者回報的數據(瀏覽/互動/留言/轉發/點擊/DM/名單/銷售,有多少收多少)。
2. 在 `LOG.md` 新增一筆當日紀錄(依範本:做了什麼實驗、賭什麼、數據、結論)。
3. 做**歸因**:這個結果驗證/推翻了 MODEL.md 哪一條假設?
4. 更新 `MODEL.md`:
   - 有效的 offer 角度 / 漏斗 / 定位 → 升信心度(理論→觀察中→已驗證),寫進「六、已驗證的賺錢模式」。
   - 連兩次無效 → 降級或淘汰該假設,從「七、實驗待辦」移除或改寫。
   - 更新「八、關鍵指標」的當前數字。
   - 版本號 +0.1,補「版本紀錄」一列。
5. 若某實驗已結案 → 從「七、實驗待辦」補下一個要測的假設(永遠保持待辦非空)。
6. 回報使用者(3 點內):今天學到什麼、模型改了什麼、明天派工方向。

## 週會模式(Re-plan,建議每 7 天一次)

目標:拉高視角,重算整盤生意,避免在錯的方向上勤奮。

1. 讀 `MODEL.md` 全文 + `LOG.md` 過去一週。
2. 對照 Hormozi 四支柱重新自問(見 MODEL.md「一」):
   - 市場對不對?(飢餓人群:痛、購買力、易觸達、成長中)
   - 報價夠不夠「無法拒絕」?用價值公式檢查分子/分母。
   - 流量的「量」夠不夠?(量能抵消運氣)
   - 轉換卡在哪一關?(內容→名單→報價→成交)
3. 產出「下週主打假設」:一句話寫死「這週要證明或推翻的一件事」。
4. 更新 MODEL.md「五、Offer 階梯」與「七、實驗待辦」,版本號 +0.1(週會用整數位,如 v1.0→v2.0 視變動大小)。
5. 用派工模式排出下週節奏。

## 原則

- 全程繁體中文、台灣用語。
- **一次只賭一個主要變因**:每個實驗要能明確歸因,不要同時改 offer + 漏斗 + 定位。
- **量先於完美**(Hormozi:量能抵消運氣):寧可粗糙地發 5 篇,不要精緻地發 1 篇再消失。
- **數據 > 感覺**:任何規則升級都要有真實數字;沒數據的想法一律標「理論」。
- 更新檔案時保持章節結構不變,只改內容。
- 行事曆事件的時間要落在 MODEL.md 記錄的黃金時段;與既有行程衝突時往後挪、不硬塞。
