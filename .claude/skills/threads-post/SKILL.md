# Threads 貼文設計 + 學習迴圈

本 skill 有兩種模式,先判斷使用者意圖:

- 使用者給了**標題/主題 + 素材資料** → 進入【生成模式】
- 使用者上傳了**貼文截圖**(或說要分析成效) → 進入【學習模式】

兩種模式都必須先完整讀取同目錄的 `MODEL.md`。

---

## 生成模式

1. 讀取 `MODEL.md`(全文)。注意:「七、已驗證模式」權重最高,其次「六、客群輪廓」,再來才是通用規則。
2. 若使用者只給標題沒給素材,直接就標題發揮,不要反問。
3. 依模型生成 **3 個版本**的內文,各用不同鉤子公式(標明用了 H 幾),結構遵守漢堡結構與字數規範(500 字內、精華 100–250 字):
   - 版本 A:模型信心度最高的鉤子
   - 版本 B:不同角度(如觀點型 vs 互動型)
   - 版本 C:實驗性(用較少驗證的公式,幫模型收集數據)
4. 每個版本後面用一行說明「這版賭的是什麼」(例:賭爭議性換留言)。
5. 結尾附上發文操作提醒(從 MODEL.md 第五章摘 2–3 條,含建議發文時段)。
6. 檢查降權地雷清單,確保沒踩雷(尤其:不放連結、不用互動誘餌句、不能有 AI 腔 —— 用口語、允許不完美)。

## 學習模式

1. 讀取截圖(Read 工具讀圖片),提取:貼文全文、瀏覽數、讚數、留言數、轉發數、發文時間(能看到多少取多少);若截圖含留言區,一併提取留言內容與語氣。
2. 讀取 `MODEL.md` 與 `CASES.md`。
3. 做**流量歸因分析**:
   - 這篇用了哪個鉤子/結構?對應模型哪條規則?
   - 數據驗證了什麼(留言比 > 讚比?哪句話引發討論?)
   - 有沒有模型沒涵蓋的新模式?
   - 從留言區推斷客群:誰在回、對什麼字眼有反應。
   - 若數據不佳,同樣分析原因 —— 失敗案例也要記錄並降級對應規則。
4. **更新 `CASES.md`**:依範本新增一筆案例(Case 編號遞增)。
5. **更新 `MODEL.md`**:
   - 鉤子公式信心度升降級(驗證有效→升級;連兩次差→降級)。
   - 新發現的模式寫入「七、已驗證模式」。
   - 客群洞察寫入「六、客群輪廓」(累積式更新,不要覆蓋舊洞察,矛盾時標註)。
   - 版本號 +0.1,並在「版本紀錄」表格新增一列。
6. 向使用者回報:這篇為什麼會(或不會)有流量(3 點內)、模型更新了什麼、下一篇建議怎麼寫。

## 原則

- 所有輸出使用繁體中文、台灣用語。
- 生成的內文要「像人隨手打的」:允許口語、破折號、換行呼吸感;禁止 markdown 標題、禁止過度工整的排比。
- 一次分析多張截圖時,每張各記一筆 Case,最後統一做一次模型更新。
- 更新 MODEL.md 時保持檔案結構不變(七個章節 + 版本紀錄),只改內容。
