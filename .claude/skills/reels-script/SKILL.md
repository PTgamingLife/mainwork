# IG Reels 短影音設計 + 學習迴圈

本 skill 有兩種模式,先判斷使用者意圖:

- 使用者給了**標題/主題 + 素材資料** → 進入【生成模式】
- 使用者給了 **reels 網址、影片截圖、或數據**(或說要分析成效) → 進入【學習模式】

兩種模式都必須先完整讀取同目錄的 `MODEL.md`。

---

## 生成模式

1. 讀取 `MODEL.md`(全文)。權重順序:「八、已驗證模式」最高 → 「七、客群輪廓」→ 通用規則(一~六章)。
2. 若使用者只給標題沒給素材,直接就標題發揮,不要反問。
3. 產出**完整可拍腳本**,包含以下欄位(用表格或分段呈現,秒數要標):
   - **選題定位**:這支影片賭哪個流量機制(完播?轉發?收藏?),一句話說明。
   - **Hook(0–3 秒)**:給 **3 個版本**,各用不同鉤子公式(標明 H 幾)。每個版本要同時給三層:口白句 + 畫面(第一幀拍什麼)+ 字卡(3–7 字大字)。
     - 版本 A:模型信心度最高的公式
     - 版本 B:不同類型(如懸念型 vs 負面警告型)
     - 版本 C:實驗性(較少驗證的公式,幫模型收集數據)
   - **主體腳本(逐段)**:每段含「秒數|口白|畫面/運鏡|字卡」。遵守 MODEL.md 的節奏規範(每 2–3 秒一個視覺變化、20–30 秒補第二鉤子)。
   - **結尾(最後 3–5 秒)**:收尾句 + CTA(依選題定位選:收藏型/轉發型/留言型)+ 循環設計(結尾畫面接回第一幀)。
   - **Caption 文案**:第一行當第二 hook,含關鍵字(SEO),1–3 個標籤。
4. 每個 hook 版本後面用一行說明「這版賭的是什麼」。
5. 結尾附上發布操作提醒(從 MODEL.md 第六章摘 2–3 條)。
6. 檢查降權地雷清單(第五章),確保沒踩雷。

## 學習模式

1. **收集這支 reels 的資料**,按順序嘗試:
   - 使用者貼網址 → 先用 WebFetch 抓公開資訊(caption、帳號);IG 常擋爬蟲,抓不到就直接問使用者要:影片文稿(或口述內容)、數據(觀看/讚/留言/轉發/收藏)、截圖。
   - 使用者給截圖 → 用 Read 讀圖,提取:畫面構成、字卡、數據、留言內容與語氣。
   - 不要卡在抓不到資料;有多少分析多少,缺的欄位在 Case 裡標「未知」。
2. 讀取 `MODEL.md` 與 `CASES.md`。
3. 做**流量歸因分析**:
   - Hook 是哪型?對應模型哪條公式?前 3 秒的口白/畫面/字卡三層各做了什麼?
   - 這支贏在哪個機制:完播(短+循環)?轉發(共鳴/實用)?收藏(教學/清單)?留言(爭議/提問)?
   - 節奏:多久一個視覺變化?有沒有第二鉤子?結尾有沒有循環設計?
   - 從留言區推斷客群:誰在回、對什麼字眼有反應。
   - 有沒有模型沒涵蓋的新模式?
   - 若數據不佳,同樣分析原因 —— 失敗案例也要記錄並降級對應規則。
4. **更新 `CASES.md`**:依範本新增一筆案例(Case 編號遞增),記下網址。
5. **更新 `MODEL.md`**:
   - 鉤子公式信心度升降級(驗證有效→升級;連兩次差→降級)。
   - 新發現的模式寫入「八、已驗證模式」。
   - 客群洞察寫入「七、客群輪廓」(累積式更新,不要覆蓋舊洞察,矛盾時標註)。
   - 版本號 +0.1,並在「版本紀錄」表格新增一列。
6. 向使用者回報:這支為什麼會(或不會)有流量(3 點內)、模型更新了什麼、下一支建議怎麼拍。

## 原則

- 所有輸出使用繁體中文、台灣用語;口白要「講得出口」的口語,不是書面文。
- 腳本欄位務必具體到「照著就能拍」:畫面欄寫清楚拍什麼(自拍講話?俯拍手部?螢幕錄影?B-roll?),不寫抽象形容詞。
- 一次分析多支 reels 時,每支各記一筆 Case,最後統一做一次模型更新。
- 更新 MODEL.md 時保持檔案結構不變(八個章節 + 版本紀錄),只改內容。
